## 2. Critical Fixtures

### 2.1 Real Hudl Export (founder-provided, BLOCKS Phase 2 validation)

**One real anonymized Hudl export** from a real game. Specifically:
- `breakdown-YYYY-WW.csv` — Hudl Assist breakdown data
- `sportscode-YYYY-WW.xml` — SportsCode-compatible XML with timestamps
- `concatenated-YYYY-WW.mp4` — concatenated clips from Hudl

**Anonymization requirements:**
- Replace player names with generic identifiers ("Player #12")
- Remove all references to the opponent school's real name in the CSV
- Strip any PII from file metadata

**Where it goes:** `tests/fixtures/real-hudl/` (git-ignored, stored separately or in encrypted Blob)

**Why it blocks Phase 2:** the reconciliation algorithm in PLAN.md §4a is only validated when we can run it against a real export. Synthetic fixtures catch known-pattern failures but cannot reproduce real-world Hudl quirks.

### 2.2 Labeled CV Benchmark Set (BLOCKS Phase 4.5 shipping)

**~100 hand-labeled plays** with ground-truth values for every CV task:

| Field | Values | Labeler notes |
|---|---|---|
| `coverage_shell` | cover_0, cover_1, cover_2, cover_3, cover_4, quarters, man_free, man_under | Determined post-snap, ~1s after ball motion |
| `pressure_type` | 4_man, 5_man, 6_man, lb_blitz, db_blitz, lb_stunt, dl_stunt | Count rushers at snap + 0.5s |
| `pressure_source` | weak_side, strong_side, middle, edge, a_gap, b_gap, c_gap | Which gap/side the free rusher came from |
| `coverage_disguise` | true / false | Does the pre-snap look differ from the post-snap coverage? |
| `cushion_depth_cb_range` | tight (0-3yd), medium (4-7yd), soft (8+yd) | For each CB on the play |
| `safety_depth_range` | box, intermediate (8-12yd), deep (13+yd) | For each safety |

**Who labels:** a coach or football analyst. This cannot be crowd-sourced — the labels require football expertise.

**Split:** 80 plays for training/prompt-tuning, 20 plays held out for CI benchmarks. The held-out set is frozen and never shown to prompt engineers during tuning.

**Where it goes:** `tests/cv-bench/labeled-plays.jsonl` — each line has `{play_id, clip_url, labels: {...}}`

**Pass criteria for Phase 4.5 shipping:**
| Task | Minimum precision on held-out set |
|---|---|
| coverage_shell | 92% |
| pressure_type | 88% |
| pressure_source | 85% |
| coverage_disguise | 90% |
| cushion_depth_cb_range | 80% |

If any task is below threshold, that task is gated behind a Vercel Flag and disabled in production until prompts improve.

### 2.3 Synthetic Hudl Exports (generated in-tree, ~20 minutes to build)

All of these are small MP4 files (10 plays, ~30 seconds each, test patterns generated via ffmpeg):

- **`good-10.csv/xml/mp4`** — 10 valid plays, matching counts, clean timestamps. Happy-path fixture.
- **`off-by-one.csv/xml/mp4`** — CSV has 10 rows, XML has 9. Reconciliation must reject.
- **`drifting-timestamps.csv/xml/mp4`** — XML max timestamp exceeds MP4 duration by >2s. Reconciliation must reject.
- **`zero-duration-segment.csv/xml/mp4`** — One XML segment has `t_end == t_start`. Reconciliation must reject.
- **`missing-column.csv/xml/mp4`** — CSV lacks the `down` column. Zod schema must reject with row number.
- **`unknown-extra-column.csv/xml/mp4`** — CSV has an unknown `CustomTag1` column. Reconciliation must warn but accept.
- **`duplicate-upload.csv/xml/mp4`** — Identical to `good-10`. Second upload returns existing game ID, does not reprocess.

**Where it goes:** `tests/fixtures/synthetic-hudl/` (checked into git since files are small)

## 3. Coverage Requirements by Phase

### Phase 0 — Scaffolding
- [ ] CI runs on PR (typecheck, lint, test all pass on empty project)
- [ ] Deploy pipeline smoke test: push to main, verify Vercel deploys, verify health endpoint responds
- [ ] Neon branching works: PR branch gets its own DB

### Phase 1 — Auth + Program Setup
- [ ] Auth middleware: unauthenticated requests redirect to Clerk sign-in
- [ ] Auth middleware: authenticated coaches can access coach routes
- [ ] Auth middleware: players cannot access coach routes (returns 403)
- [ ] Role enforcement: `assistant` role cannot publish a game plan (returns 403)
- [ ] **RLS isolation test**: create two programs with identical data, query program A's plays from program B's context, assert 0 rows returned
- [ ] Join code flow: head coach generates code, player redeems code, player's `program_id` is correct
- [ ] Duplicate join code redemption is idempotent

### Phase 2 — Hudl Ingestion
- [ ] Happy path: `good-10` fixture → 10 play rows in DB, 10 clips in Blob, status `awaiting_cv` or `ready`
- [ ] Rejection: `off-by-one` → upload rejected with clear error naming row count mismatch
- [ ] Rejection: `drifting-timestamps` → upload rejected
- [ ] Rejection: `zero-duration-segment` → upload rejected naming the bad row
- [ ] Rejection: `missing-column` → upload rejected naming the missing column and row number
- [ ] Warning: `unknown-extra-column` → upload accepted, unknown column logged
- [ ] Idempotency: upload `good-10` twice → second returns existing game, does not reprocess
- [ ] ffmpeg wrapper: clip duration matches XML segment duration ± 0.1s
- [ ] Queue fan-out: uploading `good-10` enqueues 10 clip_split jobs, each completes independently
- [ ] Partial failure: one clip_split job fails → that play marked `clip_failed`, other 9 succeed, upload is not rolled back
- [ ] **Regression test for real-world Hudl export** (blocks on §2.1 fixture)

### Phase 3 — Film Room v0
- [ ] E2E: sign in → upload → wait for processing → see play grid
- [ ] E2E: click a play → video player loads, clip plays
- [ ] E2E: apply filter chip → play grid narrows
- [ ] Film Room renders the correct tag data per play
- [ ] Signed URL delivery: browser fetches clips directly from Blob, no Function invocation

### Phase 4 — Tendency Engine
- [ ] SQL tendency query correctness (fixture-driven): given a known 20-play game, compute run rate on 1st and 10 → assert expected value
- [ ] Scouting Hub renders the opponent summary with the right numbers
- [ ] Self-scout renders the own-team predictability flags
- [ ] Every tendency card links to exactly the plays that back it (clip_count matches query result)
- [ ] Materialized tendency view refreshes on game completion
- [ ] Confidence calculation: low-sample-size tendencies show "Low confidence" label

### Phase 4.5 — Vision Intelligence Layer
- [ ] Ensemble voting: if both models agree → tag written; if disagree → tag discarded (silently, logged)
- [ ] Confidence threshold filter: agreement confidence below 0.90 → tag not written
- [ ] Prompt version: tag row carries correct `prompt_version_id`
- [ ] Reprocess with new prompt version: old tags remain, new tags write alongside, `is_active` filter picks new ones
- [ ] Per-program concurrency cap: 20 plays queued, only 5 run simultaneously
- [ ] Per-program daily spend cap: exceed → pause queue, alert fires
- [ ] Circuit breaker: one provider returns 500 on 3 consecutive calls → that provider is skipped for N minutes
- [ ] **CV benchmark gate**: each task hits its precision threshold on §2.2 held-out set
- [ ] Dead-letter: a job that fails all retries lands in a dead-letter queue with the full context

### Phase 5 — Command Bar
- [ ] 20 golden queries → expected tool calls
  - "show me every play their linebacker blitzed on 3rd and short"
  - "what's our completion rate against Cover 2 this season"
  - "add Jackson to the slot receiver group"

- [ ] Destructive actions require confirmation: "delete roster" shows a confirmation dialog before executing

- [ ] Unpublish restores editing, published documents remain unchanged until re-publish
- [ ] Dismissal of a suggestion stores the reason and the engine stops re-suggesting that pattern

### Phase 7 — Player App v0

- [ ] My Game Plan renders correctly with per-situation assignments
- [ ] Progress tab shows completion stats

### Phase 8 — Practice Builder

- [ ] Completion data writes back to coach's Roster view

### Phase 9 — The Field

## 6. Non-Negotiable Gates

1. **CI blocks on eval regressions.** If a prompt change drops any eval score below baseline, CI fails. Baseline is updated manually with an explicit commit message.
2. **CI blocks on CV accuracy regressions.** Every CV task must clear its Phase 4.5 precision threshold.
3. **RLS isolation tests run on every PR.** Cross-program reads return 0 rows, always.
4. **Typecheck is zero-tolerance.** No `any`. No `// @ts-expect-error` without a comment explaining why and a linked issue.
5. **Lint is zero-tolerance.** Biome configured to fail on warnings.
6. **Test coverage** is not enforced as a % gate (meaningless metric), but every new codepath identified in `/plan-eng-review` or `/review` sessions must have tests. Reviews enforce this.
7. **Playwright tests run against Vercel preview deployments** on every PR. If the preview deploy fails, PRs do not merge.
